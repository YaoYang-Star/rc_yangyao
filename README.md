# API 通知系统设计与实现

## 问题理解
本系统是企业内部的通用通知投递服务，接收各业务系统提交的外部HTTP通知请求，屏蔽不同外部API的差异，保证通知稳定可靠地送达目标地址，让业务系统无需关心通知投递的细节。

## 整体架构设计
```
┌─────────────┐     ┌────────────────────┐     ┌──────────────────┐
│ 业务系统    │────▶│  HTTP 接入层       │────▶│  消息队列        │
└─────────────┘     └────────────────────┘     └─────────┬────────┘
                                                         │
┌─────────────┐     ┌────────────────────┐     ┌─────────▼────────┐
│ 外部API     │◀────│  投递执行层        │◀────│  任务调度层      │
└─────────────┘     └────────────────────┘     └──────────────────┘
```

## 新增功能特性
### v1.2.0 版本更新（最新）
✅ **幂等性支持**：支持业务方传入IdempotentID，避免重复投递，保证业务逻辑正确性
✅ **批量通知接口**：新增`/notify/batch`接口，支持一次性提交最多100个通知任务，大幅提升业务方接入效率
✅ **回调通知功能**：支持配置CallbackURL，投递成功/失败后自动回调业务系统，无需业务方轮询状态
✅ **死信队列管理接口**：提供完整的死信队列管理能力：
  - 分页查询死信列表
  - 重试单个/所有死信任务
  - 删除死信任务
  - 不需要连接Redis即可运维操作
✅ **性能优化**：批量提交使用Redis Pipeline，大幅提升高并发场景下的吞吐量
✅ **错误处理增强**：回调失败自动重试3次，保证回调成功率

### v1.1.0 版本更新
✅ **核心bug修复**：修复投递时body未正确发送的严重问题，错误信息格式化错误问题
✅ **配置管理**：支持通过环境变量配置所有参数，无需修改代码
✅ **全链路追踪**：每个任务生成唯一TaskID，支持任务状态跟踪查询
✅ **安全加固**：
  - API Key鉴权，防止未授权访问
  - SSRF防护，禁止内网地址请求，避免安全风险
✅ **可观测性提升**：
  - 健康检查接口，支持监控告警
  - 任务状态查询接口，可实时查看投递进度
  - 结构化日志，全链路透传TaskID，方便排查问题
✅ **高可用增强**：
  - 优雅停机，进程退出时等待任务处理完成，避免消息丢失
  - 完善的错误处理，所有Redis操作错误都被捕获处理
✅ **部署友好**：
  - 提供Dockerfile和docker-compose.yml，一键部署
  - 支持K8s部署，健康检查接口符合云原生标准

### 核心模块
1. **HTTP接入层**：提供统一的API接口给业务系统调用，接收通知请求，做参数校验后写入消息队列
2. **消息队列**：异步解耦，缓冲流量峰值，保证请求不丢失
3. **任务调度层**：负责任务的重试调度、失败处理、延迟执行
4. **投递执行层**：实际执行HTTP请求投递，处理不同的Header/Body格式，记录投递结果

## 快速开始

### 本地运行
```bash
# 安装依赖
go mod download

# 启动Redis
redis-server

# 启动服务
go run main.go

# 提交通知请求
curl -X POST http://localhost:8080/notify \
  -H "Content-Type: application/json" \
  -H "X-API-Key: your-api-key" \
  -d '{
    "url": "https://example.com/webhook",
    "method": "POST",
    "headers": {
      "X-Custom-Header": "test"
    },
    "body": {
      "event": "user_register",
      "user_id": 12345
    }
  }'

# 查询任务状态
curl http://localhost:8080/task/{task_id}
```

### Docker部署
```bash
# 一键启动
docker-compose up -d

# 停止服务
docker-compose down
```

## 关键工程决策

### 1. 系统边界
#### 已解决的问题
- ✅ 通知请求的统一接收与格式校验
- ✅ 不同外部API的Header/Body格式适配
- ✅ 失败自动重试与退避策略
- ✅ 投递状态跟踪与查询
- ✅ 流量削峰与异步解耦

#### 不解决的问题
- ❌ 业务数据校验：通知内容的业务正确性由上游业务系统负责
- ❌ 外部API的幂等性保证：由外部API自身实现，系统仅保证投递语义
- ❌ 复杂的数据分析与报表：第一版仅保留核心日志，不做复杂统计
- ❌ 多租户隔离：初期为内部服务，不需要租户隔离能力
- **原因**：聚焦核心需求，降低第一版复杂度，快速上线验证价值

### 2. 可靠性与失败处理
#### 投递语义：至少一次（At Least Once）
- 选择原因：业务场景允许少量重复通知（如用户注册通知、付款通知），但不允许丢失
- 实现方式：消息队列ACK机制 + 投递成功后才标记任务完成

#### 失败处理策略
1. **重试策略**：失败后采用指数退避重试，初始间隔1s，最大间隔1小时，最多重试24次
2. **死信队列**：重试24次仍失败的请求进入死信队列，人工介入处理
3. **熔断机制**：单个外部API连续失败超过10次时，熔断5分钟，避免无效请求消耗资源
4. **告警通知**：死信队列有新消息时发送告警给运维人员

### 3. 技术选型
#### 选择Redis作为消息队列和任务存储
- **原因**：
  1. 轻量级，部署运维简单，适合初期小流量场景
  2. 支持延迟队列（ZSET），可以很方便实现重试调度
  3. 性能足够支撑初期QPS（万级以下）
  4. 企业内部通常已有Redis集群，不需要额外引入新的中间件
- **替代方案**：
  - 不使用中间件：用数据库表存储任务，后台线程轮询，缺点是轮询效率低，无法应对高流量
  - 使用Kafka：适合大流量场景，但部署运维复杂，对于初期小流量来说过度设计

## 未来演进方向
当流量或复杂度增长时，可以按以下方向演进：
1. **流量增长到10万QPS以上**：替换Redis为Kafka，提升消息队列吞吐量
2. **多租户需求**：增加租户管理模块，支持不同业务线的隔离和配额管理
3. **复杂统计需求**：增加数据分析模块，提供投递成功率、延迟等报表
4. **多区域部署**：增加异地多活能力，提升系统可用性
5. **智能路由**：根据外部API的位置和质量自动选择最优投递节点
