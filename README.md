# API 通知系统设计与实现

## 问题理解
本系统是企业内部的通用通知投递服务，接收各业务系统提交的外部HTTP通知请求，屏蔽不同外部API的差异，保证通知稳定可靠地送达目标地址，让业务系统无需关心通知投递的细节。

## 整体架构设计
```
┌─────────────┐     ┌────────────────────┐     ┌──────────────────┐
│ 业务系统    │────▶│  HTTP 接入层       │────▶│  消息队列        │
└─────────────┘     └────────────────────┘     └─────────┬────────┘
                                                         │
┌─────────────┐     ┌────────────────────┐     ┌─────────▼────────┐
│ 外部API     │◀────│  投递执行层        │◀────│  任务调度层      │
└─────────────┘     └────────────────────┘     └──────────────────┘
```

### 核心模块
1. **HTTP接入层**：提供统一的API接口给业务系统调用，接收通知请求，做参数校验后写入消息队列
2. **消息队列**：异步解耦，缓冲流量峰值，保证请求不丢失
3. **任务调度层**：负责任务的重试调度、失败处理、延迟执行
4. **投递执行层**：实际执行HTTP请求投递，处理不同的Header/Body格式，记录投递结果

## 关键工程决策

### 1. 系统边界
#### 已解决的问题
- ✅ 通知请求的统一接收与格式校验
- ✅ 不同外部API的Header/Body格式适配
- ✅ 失败自动重试与退避策略
- ✅ 投递状态跟踪与查询
- ✅ 流量削峰与异步解耦

#### 不解决的问题
- ❌ 业务数据校验：通知内容的业务正确性由上游业务系统负责
- ❌ 外部API的幂等性保证：由外部API自身实现，系统仅保证投递语义
- ❌ 复杂的数据分析与报表：第一版仅保留核心日志，不做复杂统计
- ❌ 多租户隔离：初期为内部服务，不需要租户隔离能力
- **原因**：聚焦核心需求，降低第一版复杂度，快速上线验证价值

### 2. 可靠性与失败处理
#### 投递语义：至少一次（At Least Once）
- 选择原因：业务场景允许少量重复通知（如用户注册通知、付款通知），但不允许丢失
- 实现方式：消息队列ACK机制 + 投递成功后才标记任务完成

#### 失败处理策略
1. **重试策略**：失败后采用指数退避重试，初始间隔1s，最大间隔1小时，最多重试24次
2. **死信队列**：重试24次仍失败的请求进入死信队列，人工介入处理
3. **熔断机制**：单个外部API连续失败超过10次时，熔断5分钟，避免无效请求消耗资源
4. **告警通知**：死信队列有新消息时发送告警给运维人员

### 3. 技术选型
#### 选择Redis作为消息队列和任务存储
- **原因**：
  1. 轻量级，部署运维简单，适合初期小流量场景
  2. 支持延迟队列（ZSET），可以很方便实现重试调度
  3. 性能足够支撑初期QPS（万级以下）
  4. 企业内部通常已有Redis集群，不需要额外引入新的中间件
- **替代方案**：
  - 不使用中间件：用数据库表存储任务，后台线程轮询，缺点是轮询效率低，无法应对高流量
  - 使用Kafka：适合大流量场景，但部署运维复杂，对于初期小流量来说过度设计

## 未来演进方向
当流量或复杂度增长时，可以按以下方向演进：
1. **流量增长到10万QPS以上**：替换Redis为Kafka，提升消息队列吞吐量
2. **多租户需求**：增加租户管理模块，支持不同业务线的隔离和配额管理
3. **复杂统计需求**：增加数据分析模块，提供投递成功率、延迟等报表
4. **多区域部署**：增加异地多活能力，提升系统可用性
5. **智能路由**：根据外部API的位置和质量自动选择最优投递节点

2. 核心代码的结构和实现建议
3. 技术选型的优缺点分析
4. 文档结构和内容组织建议

### 未采纳的AI建议
1. AI建议引入分布式追踪系统：第一版不需要，增加复杂度，后续根据需要再添加
2. AI建议支持多种消息协议（MQTT、GRPC）：初期仅需要HTTP协议，多协议支持属于过度设计
3. AI建议实现可视化管理后台：第一版优先保证核心功能，管理后台可以后续迭代

### 自主关键决策
1. 选择"至少一次"投递语义：基于对业务场景的理解，权衡可用性和一致性后做出的决策
2. 选择Redis作为核心中间件：平衡复杂度和性能，适合快速落地MVP
3. 明确系统边界，砍掉非核心功能：保证第一版可以快速交付验证价值
