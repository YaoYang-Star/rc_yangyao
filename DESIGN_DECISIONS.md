# 设计决策说明与常见问题解答

## Q1：为什么选择至少一次投递语义，而不是恰好一次？
**A：**
1. 业务场景适配：通知类场景（如注册通知、付款通知）允许少量重复，但是绝对不能丢失，"至少一次"是最符合业务需求的选择
2. 实现成本：恰好一次投递需要分布式事务、幂等校验等复杂机制，会大幅提升系统复杂度和运维成本，MVP阶段不需要
3. 责任划分：幂等性应该由外部API自身保证，通知系统只负责可靠投递，符合单一职责原则
4. 后续扩展：如果业务需要恰好一次，可以后续添加幂等ID校验功能，不需要重构核心架构

## Q2：为什么选择Redis而不是Kafka/RabbitMQ作为消息队列？
**A：**
| 对比项 | Redis | Kafka | RabbitMQ |
|-------|-------|-------|----------|
| 部署复杂度 | 极低，企业内部通常已有集群 | 高，需要ZK等配套组件 | 中等 |
| 延迟 | 亚毫秒级 | 毫秒级 | 毫秒级 |
| 吞吐量 | 万级QPS | 十万级以上QPS | 万级QPS |
| 功能支持 | 支持延迟队列，适合重试场景 | 无原生延迟队列，需要额外实现 | 支持延迟队列 |
| 运维成本 | 极低 | 高 | 中等 |

选择Redis的核心原因：
1. MVP阶段流量不大，Redis的性能完全足够
2. 原生支持延迟队列（ZSET），可以非常方便实现重试调度，不需要引入额外组件
3. 运维成本极低，不需要专门的MQ运维人员
4. 未来流量增长后，可以平滑迁移到Kafka，核心逻辑不需要大改

## Q3：如果Redis挂了怎么办？消息会不会丢失？
**A：**
1. 高可用部署：Redis采用哨兵/集群模式部署，避免单点故障
2. 持久化配置：开启RDB+AOF持久化，数据定期落盘，故障恢复后数据不丢失
3. 降级方案：接入层实现本地磁盘缓冲队列，当Redis不可用时，暂时将请求写入本地文件，Redis恢复后自动回放，保证消息不丢失
4. 限流保护：Redis故障时接入层返回系统繁忙，避免请求量过大撑爆本地磁盘

## Q4：大量重试任务会不会影响正常的投递任务？
**A：**
1. 队列隔离：主队列和重试队列完全隔离，重试任务不会抢占正常任务的资源
2. 优先级调度：正常任务的优先级高于重试任务，优先消费主队列的任务
3. 流量控制：重试任务的消费速率单独配置，避免重试流量过高影响系统稳定性
4. 熔断机制：单个外部API连续失败超过阈值时，自动熔断，避免大量无效重试

## Q5：如何避免被外部API限流？
**A：**
1. 客户端限流：针对每个外部API域名配置QPS上限，超过上限的请求自动进入延迟队列
2. 错误码识别：识别外部API返回的429限流错误码，自动延长重试间隔
3. 自适应退避：根据外部API的失败率动态调整投递速率，失败率越高，投递速率越低
4. 告警通知：当某个外部API的限流频率过高时，通知业务方优化调用频率

## Q6：为什么不实现管理后台？
**A：**
1. MVP阶段核心目标是验证投递功能的可用性，管理后台属于锦上添花的功能，不是核心需求
2. 初期运维操作可以通过命令行完成：
   ```bash
   # 查看死信队列
   redis-cli LRANGE dead_letter_queue 0 -1
   # 重试死信任务
   redis-cli RPOPLPUSH dead_letter_queue notification_queue
   # 查看队列长度
   redis-cli LLEN notification_queue
   ```
3. 后续如果有运维需求，可以快速迭代开发管理后台，核心服务不需要修改

## Q7：系统的瓶颈在哪里？如何扩容？
**A：**
1. 初期瓶颈在Redis：单Redis实例可以支撑10万QPS左右的读写，足够支撑初期业务
2. Redis瓶颈解决：
   - 水平拆分：不同业务线的队列使用不同的Redis实例
   - 升级为Redis集群，提升整体吞吐量
   - 替换为Kafka，支撑百万级以上QPS
3. 计算层瓶颈：HTTP接入层和Worker层都是无状态的，可以无限水平扩展，前面加负载均衡即可
4. 数据库瓶颈：如果后续需要存储投递历史，可以引入MySQL/Elasticsearch，水平分库分表

## Q8：如何保证消息的顺序性？
**A：**
1. 本系统默认不保证全局顺序，因为大部分通知场景不需要顺序性
2. 如果有业务需要顺序投递，可以通过业务ID哈希的方式，将同一个业务ID的消息路由到同一个队列分区，同一个分区的消息按顺序消费
3. 注意：顺序性和重试机制会有冲突，重试会导致乱序，需要顺序性的场景需要关闭自动重试，或者业务层自己处理乱序问题
