# 架构设计深度说明

## 一、核心指标设计
### SLO目标
- 投递成功率：99.9%（排除外部API自身不可用的情况）
- 平均投递延迟：< 2s（p99 < 10s）
- 系统可用性：99.95%
- 消息丢失率：0%

### 性能指标
- 单实例QPS：> 1000 TPS
- 消息队列堆积能力：> 100万条
- 重试任务调度延迟：< 1s

## 二、可观测性设计
### 1. 日志体系
```
- 访问日志：记录所有业务系统的调用请求（request_id、业务标识、参数、时间戳）
- 投递日志：记录每次投递的详细信息（目标地址、状态码、耗时、重试次数）
- 错误日志：记录系统异常、投递失败、死信队列入库等事件
- 慢查询日志：记录投递耗时超过5s的请求，用于性能优化
```
所有日志都包含全局唯一的request_id，实现全链路追踪。

### 2. 核心监控指标
```
# 业务指标
- notification_submit_total：通知提交总量
- notification_deliver_success_total：投递成功总量
- notification_deliver_fail_total：投递失败总量
- dead_letter_count：死信队列长度
- retry_queue_count：重试队列长度

# 性能指标
- deliver_duration_seconds：投递耗时分布
- queue_latency_seconds：消息在队列中的等待时间

# 系统指标
- http_request_duration：接口响应时间
- redis_command_duration：Redis操作耗时
```
支持Prometheus+Grafana可视化监控，配置告警规则：
- 投递成功率低于99%告警
- 死信队列长度>10告警
- 队列堆积超过10万条告警
- 服务实例宕机告警

### 3. 链路追踪
预留OpenTelemetry埋点，当系统复杂度提升后可以快速接入分布式追踪系统，实现从业务系统提交到外部API返回的全链路可追踪。

## 三、安全设计
1. **接口鉴权**：HTTP接入层添加API Key鉴权，只有授权的业务系统才能提交通知请求，避免恶意调用
2. **参数校验**：对URL进行校验，禁止内网地址/私有网段的请求，防止SSRF攻击
3. **Payload限制**：请求Body大小限制为1MB，避免大请求占用过多资源
4. **敏感数据脱敏**：日志中对Header中的敏感信息（如API Key、Token）进行脱敏处理
5. **HTTPS强制**：对外接口强制使用HTTPS，防止数据窃听和篡改

## 四、高可用设计
### 1. 集群部署
- 无状态HTTP接入层可以水平扩展，前面加负载均衡（Nginx/LVS）
- Worker节点可以水平扩展，通过Redis队列的天然竞争机制实现分布式任务调度
- Redis采用哨兵/集群模式部署，避免单点故障

### 2. 容灾设计
- 异地多活部署：多个地域部署集群，流量就近接入，本地故障时自动切流
- 数据持久化：Redis开启RDB+AOF持久化，消息数据不丢失
- 降级开关：当外部API大面积故障时，可以开启降级开关，暂停非核心业务的通知投递

### 3. 流量控制
- 接入层限流：每个业务系统配置QPS配额，防止单个业务的流量突增打垮整个系统
- 队列限流：当队列堆积超过阈值时，暂时拒绝新的请求，返回系统繁忙，避免雪崩
- 自适应限流：根据外部API的可用性动态调整投递速率，避免被外部接口限流

## 五、功能扩展设计
### 1. 通知模板功能
支持预设通知模板，业务系统只需要传模板ID和参数，系统自动生成请求Body，进一步简化业务系统的接入成本。

### 2. 回调通知功能
投递成功/失败后可以回调业务系统的接口，让业务系统可以感知到投递结果（可选功能，业务系统可以按需开启）。

### 3. 多协议支持
后续可以扩展支持GRPC、MQ等多种接入协议，满足不同业务系统的需求。

### 4. 幂等性保证
支持业务系统传递幂等ID，系统保证同一个幂等ID的通知只投递一次，解决重复通知的问题。

## 六、测试方案
### 单元测试覆盖
- 接口参数校验测试
- 投递逻辑测试（成功/失败场景）
- 重试策略测试（指数退避、最大重试次数）
- 死信队列测试
- 边界场景测试（空Body、超长URL、异常Header等）

### 集成测试
- 端到端投递测试：从提交请求到收到外部API的回调
- 高可用测试：杀死单个实例，验证消息不丢失、服务不中断
- 压力测试：模拟1000 TPS的流量，验证系统稳定性和性能指标
- 故障注入测试：模拟Redis故障、外部API故障，验证系统的容错能力

## 七、部署方案
### Docker部署
```dockerfile
FROM golang:1.21-alpine AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN go build -o notification-server .

FROM alpine:3.18
WORKDIR /app
COPY --from=builder /app/notification-server .
EXPOSE 8080
CMD ["./notification-server"]
```

### Kubernetes部署
- Deployment：部署无状态的接入层和worker节点
- ConfigMap：存储配置信息（Redis地址、重试次数、超时时间等）
- Secret：存储API Key等敏感信息
- Service：暴露服务供业务系统调用
- HPA：根据CPU使用率和队列长度自动扩缩容

## 八、常见问题预案
| 问题场景 | 处理策略 |
|---------|---------|
| 外部API大面积宕机 | 触发熔断，暂停投递，告警通知人工处理，恢复后自动重试积压的任务 |
| Redis集群故障 | 接入层暂时将请求写入本地磁盘文件，Redis恢复后自动回放，保证消息不丢失 |
| 队列严重堆积 | 自动扩容worker节点，提升消费速度，同时触发限流保护 |
| 死信队列告警 | 自动通知运维人员，支持后台手动重试/丢弃死信任务 |
